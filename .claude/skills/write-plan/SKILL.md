# write-plan — PR-Friendly Plan Document Skill

큰 단위의 작업(리팩토링, 신규 핵심 기능)을 시작하기 전에 `.temp/plans/` 아래에 계획 문서를 작성한다.
이 문서는 PR description의 근거가 되며, 구현 전에 리뷰어와 방향을 맞추는 데 사용된다.

---

## 1. 파일 넘버링 결정

`.temp/plans/` 디렉터리를 Glob으로 조회하여 `NNN-` 형식의 최대 번호를 찾는다.

- 정수 3자리 prefix를 가진 파일만 집계한다 (예: `006-`, `018-`).
- 서브 넘버링(예: `014-1-`, `014-2-`)은 별도 파일이 아니라 주 플랜의 하위 단계에 속한다.
  - 동일 주제의 파이프라인 분할이라면 `NNN-1-`, `NNN-2-` 형식을 사용한다.
  - 독립적인 새 작업이라면 `NNN+1` 정수 번호를 사용한다.
- 다음 번호 = `max(정수 번호) + 1`. 3자리로 0-패딩 (예: `019`).

파일명 패턴: `NNN-kebab-case-title.md`

---

## 2. 문서 구조 (6개 섹션, 고정 순서)

```markdown
# NNN. 제목 (한국어, 명사형)

## 개요

## 문제 상황

## 해결 방안

## 기대 효과

## 영향 범위

## 구현 단계
```

---

## 3. 각 섹션 작성 지침

### 개요

작업의 배경과 목적을 2~4문장으로 요약한다.

- 왜 이 작업이 필요한가?
- 어떤 목표를 달성하는가?
- 기존 방식 대비 무엇이 달라지는가?

### 문제 상황

현재 코드의 구체적인 문제를 **소제목 + 코드 블록** 형식으로 나열한다.

````markdown
### 현재 구조의 문제

```csharp
// 실제 코드 발췌 — 문제가 되는 부분을 주석으로 표시
```
````

**문제 1 — 제목**
설명.

**문제 2 — 제목**
설명.

````

- 문제는 코드 발췌와 함께 제시한다. 코드 없이 추상적으로 서술하지 않는다.
- 문제가 없으면 "현재 구현 없음"을 명시한다 (신규 기능).

### 해결 방안
단계별 설계를 **단계 N** 소제목과 코드 블록으로 제시한다.

```markdown
### 단계 1 — 신규 클래스 설계

```csharp
// 제안 코드 — 실제로 작성할 코드 형태
````

### 단계 2 — 기존 클래스 수정

````

- 코드 블록은 실제로 작성할 코드 스니펫 수준으로 구체적이어야 한다.
- 추상적인 서술만으로는 충분하지 않다.

### 기대 효과
**표** 형식으로 현재 vs 개선 후를 비교한다.

```markdown
| 항목 | 현재 | 개선 후 |
|---|---|---|
| 항목명 | 현재 상태 | 개선된 상태 |
````

3~6개 항목이 적절하다. 너무 많으면 핵심 항목만 남긴다.

### 영향 범위

세 가지 하위 항목을 반드시 포함한다.

**신규 파일 생성 전 자문 — 다음 질문에 모두 "예"일 때만 신규 파일을 만든다:**

1. 기존 파일 중 이 책임을 수용할 수 있는 파일이 없는가?
2. 이 로직을 기존 파일에 추가하면 단일 책임 원칙을 명백히 위반하는가?
3. 이 파일이 없으면 다른 파일이 이 로직을 중복 구현해야 하는가?

하나라도 "아니오"라면 기존 파일을 수정하는 방향으로 설계를 재검토한다.

```markdown
### 수정 대상 파일

- `Assets/Scripts/.../FileName.cs`
  — 변경 내용 한 줄 요약

### 신규 생성 파일
```

Assets/Scripts/.../
NewFile.cs
AnotherNewFile.cs

```

### 위험도
**낮음 / 중간 / 높음.** 이유 한 문장.
```

- 수정 대상이 없으면 "(없음)"으로 명시한다.
- 신규 파일이 없으면 "(없음)"으로 명시한다.

### 구현 단계

`### Step N — 제목` 형식으로 순서를 명시한다.

````markdown
### Step 1 — 파일 생성

`Assets/Scripts/.../NewFile.cs`를 생성한다. [구체적 내용].

```csharp
// 해당 Step에서 작성할 코드
```
````

### Step 2 — 기존 파일 수정

...

### Step N — 검증

- [검증 항목 1]
- [검증 항목 2]

```

- 마지막 Step은 항상 **검증** 단계여야 한다.
- 검증 항목은 Editor 실행, 로그 확인, 씬 테스트 등 구체적인 행동으로 기술한다.

## 4. 작업 분해 판단

작업 지시를 받으면 먼저 단일 플랜으로 작성할지, 여러 플랜으로 분해할지 판단한다.

### 분해 기준 — 다음 중 하나라도 해당하면 분해한다

- 독립적으로 머지 가능한 단위가 2개 이상 존재한다.
- 한 단위의 완료 여부가 다음 단위의 설계에 영향을 준다.
- 파일 수정 범위가 서로 다른 레이어(예: 데이터 모델 / 로직 / UI)에 걸친다.

### 분해하지 않는 경우

- 수정 파일이 3개 이하이고 단일 목적으로 묶인다.
- 분해해도 각 플랜이 독립 실행 불가능하다 (분해 실익 없음).

### 분해 시 넘버링 규칙

같은 작업에서 파생된 서브 태스크는 **주 번호-서브 번호** 형식을 사용한다.

```
NNN-1-first-subtask.md
NNN-2-second-subtask.md
NNN-3-third-subtask.md
```

- 주 번호 NNN은 기존 최대 정수 번호 + 1로 동일하게 사용한다.
- 서브 번호는 구현 선후 순서를 반영한다 (먼저 구현할 것이 낮은 번호).

---

## 5. 다중 플랜 작성 규칙

여러 플랜을 작성할 때 일관성을 보장하기 위해 다음을 따른다.

### 충돌 방지

- 동일한 파일을 두 플랜이 동시에 수정하는 경우, 한 플랜이 완료된 후의 상태를 기준으로 다음 플랜을 작성한다.
- 영향 범위 섹션에 "이 플랜은 NNN-M 완료 후 적용"을 명시한다.

### 후속 플랜 고려

- 앞선 플랜의 해결 방안에서 후속 플랜이 의존할 인터페이스(클래스명, 메서드 시그니처, 필드명)를 확정한다.
- 후속 플랜은 해당 인터페이스를 그대로 사용하여 작성한다. 앞선 플랜과 다른 설계를 쓰지 않는다.
- 각 플랜의 **개요** 첫 줄에 선행 플랜 번호를 명시한다:
  - 첫 번째 플랜: `(선행 플랜 없음)`
  - 이후 플랜: `(선행: NNN-M 완료 후 시작)`

---

## 6. 실행 절차

1. 사용자의 작업 지시를 분석하여 작업 범위를 파악한다.
2. 관련 파일을 Read/Grep으로 읽어 **실제 코드**를 기반으로 설계한다. 코드를 읽지 않고 가정으로 작성하지 않는다.
3. 분해 여부를 판단한다 (섹션 4 참조).
4. 넘버링을 결정한다 (섹션 1 참조).
5. **단일 플랜:** `.temp/plans/NNN-kebab-title.md`를 작성한다.  
   **다중 플랜:** 선후 순서대로 `NNN-1-`, `NNN-2-`, ... 를 순서대로 작성한다. 앞선 플랜을 먼저 완성한 뒤 다음 플랜을 작성한다 (인터페이스 확정 후 작성).
6. 생성한 파일 경로 목록을 사용자에게 알린다.

> 플랜 문서는 구현 전에 작성한다. 구현과 동시에 또는 구현 후에 작성하지 않는다.
```
